What is Payload CMS?

A modern content management system that separates content editing from website display, so your team can update the site without touching code.

High6 Inc. · Internal Presentation

First, what is it?

A CMS (Content Management System) is a tool that lets you create and edit the text, images, and other content on a website — without needing to write code. WordPress is a CMS you've probably encountered.

Payload is a headless CMS. “Headless” means the content-editing part (the “body”) and the website that displays the content (the “head”) are two separate systems. They talk to each other through an API, not by being bolted together in the same application.

Traditional CMS

Editing & display are glued together

WordPress runs both the admin panel and the public website in one system. Change something, and you're working inside the same codebase that visitors see.

Headless CMS (Payload)

Editing is separate from display

Payload handles content editing in its own application. The website fetches that content over an API and displays it independently. Each system can be updated or even replaced without touching the other.

For the developers

Stack Payload 3.85 + Next.js 16 App Router + MongoDB + TypeScript strict mode. Built on top of the official Payload Website Template and extended with @payloadcms/plugin-multi-tenant. Content is exposed via both REST and GraphQL APIs, tenant-scoped automatically.

Why did we use this?

apir-tayo's homepage content — FAQs, testimonials, portfolio items, pricing — used to be hardcoded directly in the website's code. Changing a single FAQ answer meant a developer had to edit code files and redeploy the entire site.

Before

Every content change needed a developer

Text and images were baked into the frontend code. Update an FAQ? Edit a TypeScript file, commit, and redeploy. Even the simplest change was a development task.

After

Content lives in an editable admin panel

Content is stored in Payload and edited through a visual admin panel. A non-developer can update FAQs, swap images, or add testimonials — no code, no deploy, no waiting.

Changing an FAQ answer used to mean editing code and redeploying.
Now it means typing in a form and hitting Save.

What can it do?

Visual content editing

Edit text, images, and page layouts through a clean admin panel — not a database, not a code editor. Think of it like a form-based editor designed for people who manage websites, not people who build them.

Media uploads with safety checks

Upload images directly to Payload. It automatically enforces file type restrictions and size limits, generates multiple sizes for different uses (thumbnail, preview, full), and keeps uploads organized in folders — so nobody accidentally uploads a 50MB raw photo to the live site.

Multi-tenant, multi-site

One Payload instance can serve content to multiple different websites. Content is organized in a clear hierarchy: a Tenant (like High6) owns one or more Sites (like apir-tayo). Each site's content is scoped and isolated — editing apir-tayo can't accidentally change another site. Add a new site later without rebuilding anything.

Content

API-first

All content is exposed through an API (both REST and GraphQL) that any frontend can fetch from. The website doesn't need to know how Payload stores data internally — it just asks “what's the latest content for this page?” and gets back structured data. The same API could power a mobile app or a third-party integration later.

For the developers

The REST API is available at /api/:collection with full query support (filters, pagination, depth, locale). GraphQL is at /api/graphql. Both are automatically tenant-scoped — queries only return content belonging to the current tenant context. The multi-tenant plugin layers access control on top of Payload's existing auth system, so there's no separate permission layer to maintain.

How does it actually work?

At a high level, there are two separate systems working together: Payload (where content is edited) and apir-tayo (the live website visitors see). They communicate over an API, not by sharing code or a database.

1

Edit in Payload

Someone edits content in the admin panel — changes an FAQ, swaps a hero image, adds a testimonial. The change is saved to MongoDB.

2

API delivers content

apir-tayo's website asks Payload's API: “What's the current content?” Payload returns it as structured JSON data.

3

Visitors see it

The website renders the content into the live page. Visitors see the update without knowing anything changed behind the scenes.

Note on timing: There's a brief delay (up to 60 seconds) between an edit in Payload and it appearing on the live site. This is intentional — the site keeps a locally cached copy of the content so it loads instantly for every visitor, rather than asking Payload for fresh content on every single page view. The cache re-validates automatically; no manual step needed. (A “publish instantly” button can be added later if the team decides it's necessary.)

What's next

This is the starting point — not the finish line. The same approach that moved FAQs and testimonials into Payload can be applied to every content section on the site.

Foundation Done

Payload CMS is running with multi-tenant support, content collections (Pages, Posts, Media, Categories), and REST + GraphQL APIs. apir-tayo content is seeded and served through the Tenant → Site → Content structure.

More content sections Planned

Migrate additional apir-tayo sections (Hero, How It Works, portfolio detail pages, pricing tiers) into Payload the same way — one section at a time, each one becoming editable through the admin panel.

AI Agent On hold — team discussion needed

The core concept is already implemented and working: a /api/agent endpoint accepts plain-language instructions (“Change the pricing page title to Our Plans”), parses them into CMS actions, and executes them against Payload's API. Guardrails define exactly which fields and collections the agent is allowed to modify — no deleting, no tenant reassignment, no touching anything outside the approved scope. This means both developers and clients could configure their sites by typing instructions instead of navigating the admin panel. This feature is intentionally paused for team discussion before expanding beyond its current proof-of-concept scope.
